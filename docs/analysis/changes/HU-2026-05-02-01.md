- **Responsable:** Antigravity

- **Referencia técnica:** Relajamiento de optimización concurrente para endpoints tipo POST.

---

## 2. Objetivo de la HU

Relajar y eliminar el requisito del `ConcurrencyToken` de todos los request DTOs y Comandos que se utilizan para operaciones de creación (POST).

### Resumen
Originalmente, los endpoints de creación (POST) exigían enviar el `ConcurrencyToken` como si fueran actualizaciones, introduciendo complejidad innecesaria en el cliente (frontend). El objetivo de esta historia es delegar el control de concurrencia al frontend durante el flujo de creación inicial y simplificar las validaciones, permitiendo así operaciones POST verdaderamente atómicas e independientes del estado previo del token.

---

## 3. Alcance implementado

### Incluye
- Eliminación del campo `ConcurrencyToken` en los records de Request (contratos de API).
- Eliminación del argumento `ConcurrencyToken` de las definiciones de `ICommand` correspondientes a la creación de recursos (Application Layer).
- Eliminación de reglas explícitas de validación (`RuleFor(x => x.ConcurrencyToken)`) en los Validators asociados a creaciones.
- Supresión de validaciones internas del token en las cargas transaccionales de los Handlers (inyectando `Guid.Empty`).
- Limpieza de código muerto inalcanzable (`if(false)`).
- Limpieza y mantenimiento correctivo de la capa de Unit Tests para asegurar su acoplamiento al nuevo esquema.

### No incluye
- Retiro del `ConcurrencyToken` para operaciones de actualización (PUT/PATCH), las cuales siguen manteniendo concurrencia optimista estricta.

### Cambios funcionales
- El cliente ahora puede disparar POSTs de inserción (como nuevas Cuentas Bancarias, Elementos Educativos, etc.) sin depender de saber en qué estado exacto quedó el token principal del expediente en la operación anterior.

### Componentes modificados
- `Command / Query / Handler`: Comandos `Add...Command` y sus manejadores.
- `Validator`: `Add...CommandValidator` (retiro de reglas).
- `Controller / Endpoint`: Eliminación de mapeos del token desde la solicitud JSON hacia los comandos.

### Resumen técnico
Las operaciones de inserción han sido independizadas del control de versiones optimista. Internamente, el backend elude la comprobación de versión para métodos definidos como inyecciones (Add), pero sigue protegiendo a la entidad padre contra daños asíncronos o bloqueos.
