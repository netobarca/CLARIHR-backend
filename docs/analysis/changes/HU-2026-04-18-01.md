- **Referencia tecnica:** `docs/analysis/current-state/security-analysis.md`

## 2. Objetivo de la HU

Cerrar el primer paquete de riesgos criticos antes de abordar performance y refactors mayores: tenant isolation fail-closed, gobernanza de bypasses de filtros tenant, redaccion de PII en auditoria, headers anti-cache para `/api` y guardrails de upload documental.

## 3. Alcance implementado

### Incluye
- Filtro global tenant `fail-closed` para entidades `ITenantScopedEntity`.
- Comentario obligatorio en cada `IgnoreQueryFilters()` intencional y prueba de gobernanza.
- Redaccion de PII en `AuditSanitizer` y remocion de secretos/tokens como antes.
- `Cache-Control: no-store`, `Pragma: no-cache` y `Expires: 0` para toda ruta `/api` en Core API y Backoffice API.
- Limite de `10 MiB` y validacion de extension, MIME y firma basica para upload de documentos de expediente.
- Nuevos errores `PERSONNEL_FILE_DOCUMENT_TOO_LARGE` (`413`) y `PERSONNEL_FILE_DOCUMENT_CONTENT_TYPE_UNSUPPORTED` (`400`).

### No incluye
- Migracion o limpieza de logs historicos.
- Antivirus, streaming storage o procesamiento asincrono de documentos.
- Optimizacion de exportes, N+1, transacciones largas o refactor de controllers grandes.

## 4. Impacto funcional

Los usuarios mantienen el mismo contrato de upload de documentos, pero archivos no permitidos o mayores a `10 MiB` se rechazan antes de leer el binario completo. Las respuestas `/api` dejan de ser cacheables por navegadores o proxies compartidos.

### ¿Requiere actualizacion de flujo de negocio?
