- **Título:** Catálogos internos globales para requisitos de Job Profiles
- **Módulo:** Core API / Job Profiles / Internal Catalogs
- **Fecha de cierre:** 2026-04-03
- **Estado:** Completada
- **Responsable:** Codex
- **Referencia funcional:** Requerimiento para controlar requisitos de `Job Profiles` con catálogos internos globales, búsqueda por similitud y alta controlada por usuario autenticado
- **Referencia técnica:** `InternalCatalogValue`, `InternalCatalogAdministration`, `AccountInternalCatalogsController`, `JobProfileAdministration`

---

## 2. Objetivo de la HU

### Resumen
Agregar un subsistema reusable de catálogos internos globales para que el frontend pueda descubrir cómo renderizar requisitos de `Job Profiles`, buscar sugerencias reutilizables y permitir altas asistidas por similitud sin depender de tenant, `companyId` o mantenimiento manual diario del equipo.

---

## 3. Alcance implementado

### Incluye
- Nuevo agregado global `InternalCatalogValue` con normalización fuerte, metadatos de uso y migración EF propia.
- Manifest público `GET /api/account/internal-catalogs?context=job-profile.requirements` con `renderType`, `catalogKey`, `allowCreate` y `minQueryLength`.
- Búsqueda global `GET /api/account/internal-catalogs/{catalogKey}/values` con similitud mínima `0.70`.
- Alta global `POST /api/account/internal-catalogs/{catalogKey}/values` con reuse exacto, creación real o rechazo `409` cuando hay similitud `>= 0.90`.
- Autopoblado del catálogo global desde `Create/UpdateJobProfile` para requisitos `Education`, `Knowledge` y `Certification`.
- Persistencia de auditoría global para altas aceptadas en el catálogo interno, tanto manuales como disparadas desde `JobProfiles`.
- Cobertura unitaria e integración dirigida para normalización, thresholds, conflicto por similitud, acceso autenticado sin tenant y autopoblado desde perfiles.

### No incluye
- Migración de los `JobCatalogs` tenant-scoped existentes al nuevo modelo global.
- Moderación manual, rate limiting o workflows de aprobación para altas globales.
- Superficie `select` activa en el contexto `job-profile.requirements`; el contrato la soporta, pero esta primera versión no la usa.

## 4. Impacto funcional

### Cambios funcionales
- El frontend puede descubrir qué tipo de input renderizar para cada requisito de `Job Profiles` sin hardcodear reglas.
- `Education`, `Knowledge` y `Certification` ya no dependen solo de texto libre; reutilizan sugerencias globales y pueden crecer con nuevos valores validados por similitud.
- `Experience` y `Other` siguen siendo texto libre y no contaminan el catálogo global.

### Resumen técnico
La solución separa el nuevo catálogo global del `JobCatalog` tenant-scoped actual, publica un registry de definiciones en código y apoya la búsqueda en `pg_trgm` con índice `GIN` sobre `normalized_value`. Los writes globales quedan autenticados, no tenant-scoped, auditados en `PlatformAuditLog` y reutilizables tanto por el endpoint directo como por el flujo de `JobProfiles`.

## 7. Cambios en datos y persistencia

### Cambios realizados
- Tabla nueva `internal_catalog_values`
- Índice único `catalog_key + normalized_value`
- Índice de búsqueda `GIN` con `pg_trgm` sobre `normalized_value`
- Persistencia de uso (`usage_count`, `last_used_at_utc`)

### Scripts o migraciones relacionados
- `src/CLARIHR.Infrastructure/Persistence/Migrations/20260403101500_AddInternalCatalogValues.cs`

### Consideraciones
La tabla nueva es global, no tenant-scoped, y convive con los catálogos tenant-scoped existentes. El catálogo interno depende de normalización consistente en aplicación y del índice trigram para ranking y rechazo de casi duplicados.
