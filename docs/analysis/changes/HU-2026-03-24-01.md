# HU-2026-03-24-01 — Agregar rol tenant-scoped al JWT

- **Módulo:** Auth / Account companies
- **Fecha de cierre:** 2026-03-24
- **Estado:** Completada
- **Responsable:** Codex
- **Referencia funcional:** Requerimiento de incluir el rol del usuario logueado dentro del `access token`
- **Referencia técnica:** `src/CLARIHR.Infrastructure/Auth/JwtTokenService.cs`

---

## 2. Objetivo de la HU

Incluir en el `access token` el rol del usuario autenticado para el tenant efectivo del token, de forma que el backend y los clientes tengan acceso directo al contexto de rol emitido.

### Resumen
Se agregó el claim `role` al JWT cuando el token queda asociado a un tenant. El valor se resuelve desde la membresía del usuario en la compañía emitida y usa el nombre normalizado del rol para alinearse con la autorización interna.

---

## 3. Alcance implementado

### Incluye
- Resolución del rol tenant-scoped desde `IUserCompanyRepository`.
- Emisión del claim `role` en `login`, `refresh` y `switch active company` cuando el token lleva tenant.
- Pruebas unitarias e integración para validar tenant y rol del JWT.
- Actualización de documentación viva del flujo de autenticación y referencia técnica.

### No incluye
- Cambios de contrato HTTP en el body de `AuthResponse`.
- Nuevos permisos o modificaciones al modelo RBAC.
- Cambios en `openapi.yaml`, porque el token sigue expuesto como string opaco.

---

## 4. Impacto funcional

### Cambios funcionales
- Los `access tokens` emitidos para un tenant ahora incluyen el claim `role`.
- Un usuario sin compañía activa sigue recibiendo token sin `tid` ni `role`.
- Al cambiar de compañía activa, el nuevo token cambia tanto `tid` como `role`.

### Flujo afectado
- Acceso y autenticación
- Cambio de compañía activa

### Resumen técnico
`JwtTokenService` ahora construye los claims desde un helper común que, cuando existe tenant, añade `tid` y `role`. La resolución del rol usa el repositorio de membresías y hace `IgnoreQueryFilters()` sobre `IamRoles` para soportar correctamente el flujo de `switch` aun cuando el request actual viene autenticado con otro tenant.

## 12. Riesgos, limitaciones y pendientes

### Riesgos identificados
- Cualquier consumidor que lea el claim `role` debe asumir que el valor es el nombre normalizado del rol, no el nombre de display.

### Limitaciones actuales
- El token sigue exponiendo un solo rol tenant-scoped, el asociado a la membresía resuelta para el tenant emitido.
