# HU-2026-04-23-01 - Resincronizacion de permisos owner

## Resumen

Se reforzo el provisioning owner para que el usuario creador de una compania quede vinculado al rol `Admin de Empresa` con todos los permisos tenant-scoped existentes. El cambio tambien repara owners ya provisionados cuando el flujo detecta una compania primaria, evitando drift cuando el catalogo de permisos crece despues del seed inicial.

## Cambios

- Se centralizo el catalogo default de permisos owner en `OwnerPermissionCatalog`.
- `CompanyProvisioningService` ahora asegura permisos default faltantes, sincroniza el rol `Admin de Empresa` contra todos los permisos del tenant y asegura la asignacion del owner a ese rol.
- `ProvisionCompanyForUserCommand` ejecuta esa resincronizacion cuando el usuario ya tiene compania primaria.
- `DevSeedService` dejo de hacer skip seco cuando `dev@clarihr.local` ya existe y ahora sincroniza RBAC owner.
- No hubo cambios de schema.

## Validacion

- `dotnet build`
- `dotnet test tests/CLARIHR.Application.UnitTests/CLARIHR.Application.UnitTests.csproj --filter "FullyQualifiedName~ProvisionCompanyForUserCommandHandlerTests|FullyQualifiedName~CompanyUserManagementTests|FullyQualifiedName~JwtTokenServiceTests|FullyQualifiedName~AccountCompanyAdministrationTests|FullyQualifiedName~PositionSlotAdministrationTests"`
- Verificacion operativa de `dev@clarihr.local`: tenant con `49` permisos, rol admin con `49` grants y asignacion owner-admin activa. El backfill idempotente no inserto filas porque la base ya estaba alineada.
