---

## 2. Objetivo de la HU

### Resumen
Separar la administración global de plataforma del API core de RH, cerrar la brecha de seguridad basada en `PlatformAdminEmails`, formalizar la relación entre suscripciones y planes comerciales, y dejar un backoffice API propio para operar planes y suscripciones empresariales.

---

## 3. Alcance implementado

### Incluye
- Nueva `CLARIHR.Backoffice.Api` con auth propia y rutas `/api/platform/*`.
- Autorización global basada en `PlatformOperator` persistido, con roles `Admin` y `ReadOnly`.
- Auditoría durable global mediante `PlatformAuditLog`.
- CRUD de `CommercialPlan` movido al backoffice.
- Endpoints backoffice para consultar y reemplazar suscripciones empresariales.
- Relación formal de `CompanySubscription` y `PlanEntitlement` con `CommercialPlan`.
- Snapshot histórico de precios en `CompanySubscription`.
- Provisioning inicial resuelto contra el plan formal `FREE`.
- Eliminación del bypass `platform_admin` y de la allow-list `PlatformAdminEmails` del runtime.

### No incluye
- Billing real, invoices, descuentos, add-ons ni versionamiento comercial.
- Autoservicio para administrar `PlatformOperator` desde UI.
- Corrección de fallas legacy no relacionadas del suite completo.

---

## 4. Impacto funcional

### Cambios funcionales
- La administración global de planes y suscripciones ya no vive en la API core de RH.
- El backoffice autentica operadores globales con tokens `platform` separados de los tokens `core`.
- Las empresas ahora tienen una suscripción formal ligada a `CommercialPlan`, con historial al reemplazar el plan activo.

## 12. Riesgos, limitaciones y pendientes

### Riesgos identificados
- La auditoría tenant-scoped de RRHH aún persiste demasiada PII en payloads completos.
- El filtro tenant global continúa en modo `fail-open`.

### Limitaciones actuales
- El primer `PlatformOperator` sigue dependiendo de bootstrap operativo por comando.
- No existe aún CRUD autoservicio para operadores de plataforma.

### Pendientes
- Agregar pruebas explícitas de auditoría durable para todo el CRUD de `CommercialPlan`.
- Implementar controles anti abuso en auth.
- Evaluar automatización de export/versionado de OpenAPI.

## 13. Verificación funcional y técnica

### Pasos de validación
1. Intentar login en `/api/platform/auth/login` con un usuario local sin `PlatformOperator` y confirmar `403`.
2. Hacer login con un `PlatformOperator` activo y confirmar que el JWT devuelve `client_type=platform` y no incluye `tid`.
3. Crear o modificar un plan en `/api/platform/commercial-plans` y confirmar que la operación requiere token `platform`.
4. Reemplazar la suscripción activa de una empresa con `PUT /api/platform/companies/{companyPublicId}/subscription`.
5. Confirmar que queda exactamente una suscripción activa y al menos una fila histórica cancelada para la empresa.

### Resultado esperado
La administración global de planes y suscripciones queda fuera del core tenant-scoped, protegida por el backoffice API, con autorización persistida, auditoría durable y relación formal entre suscripciones y planes comerciales.
