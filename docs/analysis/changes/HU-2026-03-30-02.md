# HU-2026-03-30-02 — Cierre de implementación

## 1. Información general

- **Código HU:** HU-2026-03-30-02
- **Título:** OrgUnits expone referencia compacta del padre en lecturas
- **Módulo:** Core API / OrgUnits
- **Fecha de cierre:** 2026-03-30
- **Estado:** Completada
- **Responsable:** Codex
- **Referencia funcional:** Ajuste puntual solicitado para que listados y detalle de `OrgUnits` permitan renderizar y editar la relación padre sin lookup adicional desde frontend
- **Referencia técnica:** `OrgUnitResponse`, `OrgUnitRepository`, `ApiIntegrationTests`

---

## 2. Objetivo de la HU

### Resumen
Extender la respuesta principal de `OrgUnits` para incluir una referencia compacta de la unidad padre (`id`, `code`, `name`) sin eliminar `parentId`, de modo que el frontend pueda mostrar tablas y formularios de edición sin hacer lecturas adicionales por cada fila.

---

## 3. Alcance implementado

### Incluye
- `OrgUnitResponse` ahora expone `parent` ademas de `parentId`.
- `search` y `get by id` reutilizan el `LEFT JOIN` ya existente hacia la unidad padre para proyectar la nueva referencia sin `Include` ni queries adicionales.
- Se agregaron pruebas de integración para validar la referencia del padre en `create`, `search` y `get by id`.
- Se actualizó la referencia humana del endpoint para reflejar el nuevo shape observable.

### No incluye
- Cambios en reglas de negocio, permisos, auditoría o tenant scope del módulo.
- Cambios en `tree`, `graph`, exportes o payloads de escritura.
- Reestructuración del suplemento manual `openapi.yaml`, que aún no documenta `OrgUnits`.

---

## 4. Impacto funcional

### Cambios funcionales
- El cliente ya no necesita resolver el nombre del padre con una llamada adicional por fila.
- Los flujos de edición pueden reutilizar `parent.id` para precargar o reconciliar la selección actual del padre.

### Flujo afectado
- Administración operativa del organigrama
- `GET /api/v1/companies/{companyPublicId}/org-units`

## 6. Cambios en API

### Endpoints modificados
- `GET /api/v1/companies/{companyPublicId}/org-units` — ahora incluye `parent` con `id`, `code` y `name` cuando la org unit tiene padre.
- `GET /api/v1/org-units/{publicId}` — ahora incluye `parent` con `id`, `code` y `name` cuando la org unit tiene padre.
- `POST /api/v1/companies/{companyPublicId}/org-units` — la respuesta `201` ahora refleja el mismo `OrgUnitResponse` enriquecido.
- `PUT /api/v1/org-units/{publicId}` — la respuesta `200` hereda el mismo contrato enriquecido.
- `PATCH /api/v1/org-units/{publicId}/move` — la respuesta `200` hereda el mismo contrato enriquecido.
- `PATCH /api/v1/org-units/{publicId}/activate` — la respuesta `200` hereda el mismo contrato enriquecido.
- `PATCH /api/v1/org-units/{publicId}/inactivate` — la respuesta `200` hereda el mismo contrato enriquecido.
